\begin{frame}{Fuente y enfoque}
  \begin{itemize}
    \item \textbf{Documento base}: \textit{Big Data Scenario-based Solution Real-Time Stream Processing}.
    \item \textbf{Libro}: `huawei-2022.pdf`.
    \item \textbf{Idea central}: extraer valor de datos que pierden utilidad si no se procesan en segundos o milisegundos.
    \item \textbf{Contexto}: comercio electronico, monitoreo, riesgo financiero, IoT y servicios digitales.
  \end{itemize}
  \vspace{0.2cm}
  \begin{block}{Pregunta guia}
    Como se construye una arquitectura capaz de capturar, mover, procesar y accionar datos en flujo con baja latencia y alta confiabilidad.
  \end{block}
\end{frame}

\begin{frame}{Que es el procesamiento en tiempo real}
  \begin{block}{Definicion del capitulo}
    El flujo general es: generacion de datos, recoleccion en tiempo real, cache o almacenamiento temporal, computo en tiempo real, implementacion, visualizacion y analisis.
  \end{block}
  \begin{itemize}
    \item La velocidad esperada es del orden de segundos o milisegundos.
    \item El objetivo no es solo almacenar, sino activar decisiones mientras el evento aun es relevante.
    \item La latencia pasa a ser una variable de negocio, no solo tecnica.
  \end{itemize}
\end{frame}

\begin{frame}{Por que importa}
  \begin{itemize}
    \item El valor de algunos datos cae muy rapido despues de que ocurre el evento.
    \item En escenarios como fraude, recomendacion o alertas, llegar tarde equivale a perder la oportunidad.
    \item El \textbf{insight temprano} permite reaccion automatica, bloqueo, personalizacion o ajuste operacional.
  \end{itemize}
  \vspace{0.25cm}
  \begin{block}{Mensaje del capitulo}
    La clave del procesamiento en tiempo real es transformar datos en accion antes de que su valor se degrade.
  \end{block}
\end{frame}

\begin{frame}{Relacion con otras soluciones Big Data}
  \begin{itemize}
    \item \textbf{Offline batch}: procesa grandes volumenes historicos con mas tolerancia a la latencia.
    \item \textbf{Real-time retrieval}: prioriza consulta rapida de informacion ya cargada.
    \item \textbf{Real-time stream processing}: analiza eventos continuos y dispara acciones inmediatas.
    \item \textbf{Migracion y respaldo}: complementan la plataforma, pero no resuelven la decision en vivo.
  \end{itemize}
  \vspace{0.2cm}
  \begin{block}{Diferencia clave}
    Stream processing trabaja sobre datos en movimiento; batch processing trabaja sobre datos ya acumulados.
  \end{block}
\end{frame}

\begin{frame}{Caso de uso del capitulo: fraude con tarjetas}
  \begin{enumerate}
    \item El cliente inicia una transaccion.
    \item El canal envia el evento en tiempo real.
    \item El motor de fraude recibe, limpia y enriquece los datos.
    \item Se aplican reglas de negocio y analitica.
    \item Se devuelve una decision: aprobar, bloquear o congelar tarjeta.
  \end{enumerate}
  \vspace{0.2cm}
  \begin{itemize}
    \item El sistema combina deteccion, intervencion y respuesta operacional.
    \item Un retraso pequeno puede traducirse en perdida economica directa.
  \end{itemize}
\end{frame}

\begin{frame}{Rol de las tecnologias del capitulo}
  \begin{itemize}
    \item \textbf{Flume}: recoleccion, agregacion y transporte confiable de logs y eventos.
    \item \textbf{Kafka}: mensajeria distribuida, alta tasa de escritura y modelo publish-subscribe.
    \item \textbf{Flink}: procesamiento de streams con baja latencia y orientado a estado.
    \item \textbf{Structured Streaming}: alternativa basada en Spark para pipelines en tiempo real.
    \item \textbf{Redis}: apoyo para respuestas rapidas, cache y servicios online.
  \end{itemize}
\end{frame}

\begin{frame}{Flume como puerta de entrada}
  \begin{itemize}
    \item Arquitectura basica: \textbf{Source -> Channel -> Sink}.
    \item \textbf{Source}: recibe o genera eventos.
    \item \textbf{Channel}: cola temporal con distintos niveles de persistencia.
    \item \textbf{Sink}: envia al siguiente salto o destino final.
  \end{itemize}
  \vspace{0.25cm}
  \begin{block}{Capacidades avanzadas}
    Interceptors para enriquecer eventos, selectors para enrutar y sink processors para balanceo o failover.
  \end{block}
\end{frame}

\begin{frame}{Kafka y el desacoplamiento del flujo}
  \begin{itemize}
    \item Kafka funciona como sistema distribuido de mensajes con \textit{topics}, \textit{brokers} y consumidores.
    \item Permite absorber picos, desacoplar productores de consumidores y conservar orden dentro de particiones.
    \item Los consumidores usan \textit{offsets} para controlar lectura y relectura.
    \item Esto facilita monitoreo, analitica, data warehouse y otros servicios sobre la misma corriente de datos.
  \end{itemize}
\end{frame}

\begin{frame}{Desafios tecnicos}
  \begin{itemize}
    \item evitar perdida o duplicacion de eventos
    \item manejar picos de trafico y cambios abruptos de carga
    \item coordinar baja latencia con consistencia y tolerancia a fallos
    \item integrar reglas de negocio con modelos analiticos en linea
    \item gobernar permisos, aislamiento y uso eficiente de recursos
  \end{itemize}
  \vspace{0.2cm}
  \begin{block}{A gran escala}
    El problema no es solo procesar rapido, sino sostener esa rapidez cuando crecen datos, usuarios y servicios conectados.
  \end{block}
\end{frame}

\begin{frame}{Conclusiones}
  \begin{itemize}
    \item El capitulo presenta el stream processing como una solucion para decisiones de alto valor y baja latencia.
    \item La arquitectura combina captura, cola, computo y accion operacional.
    \item Flume y Kafka organizan el movimiento del dato; Flink y Structured Streaming ejecutan la inteligencia en linea.
    \item Los requisitos mas importantes son velocidad, throughput, confiabilidad y escalabilidad horizontal.
    \item En aplicaciones reales, el exito depende tanto de la arquitectura como de la integracion con el negocio.
  \end{itemize}
\end{frame}
