Concept of Real-Time Stream Processing


Real-time stream processing generally refers to real-time rapid data analysis to
trigger next-step actions. Real-time data analysis has high requirements on the
processing speed. In addition, due to the large amount of data, the requirements on
CPU and memory are high. In comparison, not much storage capacity is required
because the data does not need to be stored in most cases. Real-time processing is
generally implemented through Structured Streaming or Flink tasks.

Page 30

Copyright © 2020 Huawei Technologies Co., Ltd. All rights reserved.

Requirements for the Real-Time Data
Processing System (1)








Page 31

High processing speed: The end-to-end processing needs to be completed in seconds. Data
collection and processing on the stream processing platform must be completed within 1
second. For example, a risk control project requires that a single data record be processed
within seconds and the transactions per second (TPS) of a single node be greater than
2,000.
High throughput: A large number of data records need to be received and processed in a
short time, so the throughput needs to reach tens of megabytes per second per node.
Strong shockproof: The data cache mechanism must be provided to cope with the
problem when service data generated by the data source suddenly reaches the peak value.
High reliability: When a network or software fault occurs, the system must ensure that no
data record is lost and the data processing is not missing or repeated.

Copyright © 2020 Huawei Technologies Co., Ltd. All rights reserved.

Requirements for the Real-Time Data
Processing System (2)








Page 32

Horizontal expansion: When the system processing capability reaches the bottleneck, the
processing performance can be improved by adding nodes.
Support for multiple data sources: Data sources such as network streams, files, database
tables, and Internet of Things (IoT) are supported. For the file data source, incremental data
can be loaded.
Data permission and resource isolation: Data permission control is required for message
processing and stream processing so that different jobs and users can access and process
different messages and data. Resource control and isolation must be implemented between
multiple stream processing applications to prevent resource preemption.
Third-party tool interconnection: Interconnection with third-party rule engines, decision
systems, and real-time recommendation systems is supported.

Recycle Bin Mechanism of HDFS


In HDFS, files are moved to the recycle bin when they are deleted. Files in the recycle bin can
be used to quickly restore deleted files.



You can set a time threshold (unit: minute). When the storage duration of files in the recycle
bin exceeds the threshold or the recycle bin is cleared, the files are permanently deleted and
the occupied data blocks are released.



Trash of the Hadoop recycle bin is disabled by default. To enable it, modify the core-site.xml
configuration file.
<property>
<name>fs.trash.interval</name>
<value>1440</value>
</property>

Introduction to Spark


Spark is a memory-based distributed batch processing system. It splits tasks and
allocates them to multiple CPUs for processing. Intermediate products (calculation
results) generated during data processing are stored in the memory, reducing I/O
operations on disks and greatly improving the data processing speed, it has
advantages in data processing and data mining.

Page 49

Copyright © 2020 Huawei Technologies Co., Ltd. All rights reserved.

Application Scenarios of Spark


Data processing: Spark can process data quickly and is fault tolerant and scalable.



Iterative computation: Spark supports iterative computation to meet the requirements of
complex data processing logic.



Data mining: Spark can perform complex data mining and analysis on the basis of massive
data. Spark supports multiple data mining and machine learning algorithms.



Stream processing: Spark supports seconds-level latency stream processing and supports
multiple external data sources.



Query analysis: supports SQL query analysis, provides domain-specific language (DSL) to
facilitate structured data operations, and supports multiple external data sources.

Sqoop Principles (1)


Sqoop import principle:


The split-by parameter must be specified during import of Sqoop. Sqoop splits a region
based on the value of split-by and allocates the split region to different maps.



Each map processes the value of a row obtained from the database and writes the value
to HDFS.



In addition, split-by has different split methods based on different parameter types. For
example, for a simple int type, Sqoop obtains the maximum and minimum split-by field
values, and then determines the number of areas to be split based on the input nummappers.

Hadoop.



imported by a map task.

HBase Cache Mechanism


HBase provides two types of cache structures: MemStore and BlockCache.


MemStore: HBase data is written to HLogs and MemStores at the same time. After certain
conditions are met, data in MemStores is flushed to disks, which improves the write and
read performance of HBase.



BlockCache: HBase caches the block blocks found in a file search to the cache so that the
same request or adjacent data search requests can be directly obtained from the memory,
avoiding I/O operations.

HBase Performance Optimization Table Design


Region: Multiple regions are created in advance. When data is written to HBase, data load balancing is
performed in the cluster based on the region partition corresponding to the RowKey.



RowKey: row key. If the actual service requirements are met, the shorter the length, the better.
Considering hashability (continuous row keys may cause load imbalance), Hash storage may be
implemented by using negation or Hash.



Column Family: Do not define too many column families in a table. Otherwise, when a column cluster
is flushed, the adjacent column families will be flushed due to the association effect, resulting in more
I/Os.



Max Version: the maximum number of versions. If you only need to save the data of the latest version,
you can set Max Version to 1.



Time To Live: data lifetime, in seconds. For example, if only the data of the latest two days needs to be
stored, you can set the life time to 2 x 24 x 60 x 60.

Elasticsearch Performance Optimization Shard Replica Policy


The number of shards cannot be adjusted once an index is created. In Elasticsearch,
a shard corresponds to a Lucene index for storing data. Reading and writing the
Lucene index occupy a large number of system resources. Therefore, the number of
shard replicas needs to be properly configured.


The number of shards cannot exceed three times of the number of nodes. A smaller
number of shards can achieve better performance.



You are advised to set the number of copies to 1. Excessive copies require more storage
space.



The maximum size of a shard cannot exceed the maximum JVM heap size recommended
by Elasticsearch, 32 GB.

Basic Concepts - Channel (1)



Kafka - Broker


Broker: Indicates the cache proxy. One or multiple servers in a Kafka cluster are called Brokers.



To reduce the number of disk write times, the Broker temporarily buffers messages. When the number
of messages reaches a specified threshold, the Broker flushes the messages to the disk, reducing the
number of disk I/O invoking times.



Stateless mechanism of Broker:


Broker does not have the copy mechanism. Once a Broker breaks down, all messages of the broker are
unavailable.



Broker does not save the subscriber status, which is saved by the subscriber.



Statelessness makes it difficult to delete messages (because the messages that may be deleted are being
subscribed to). Kafka uses the time-based service level agreement (SLA) to ensure service quality. Messages are
deleted after being stored for a certain period (usually seven days).



The message subscriber can rewind back to any position to consume messages again. When the subscriber is
faulty, the minimum offset (id) can be selected to read the consumed messages again.

Continuous Processing of Stream Data


Unbounded stream: Only the start of a defined stream is defined. Data sources generate data endlessly. The data of unbounded
streams must be processed continuously. That is, the data must be processed immediately after being read. You cannot wait until all
data arrives before processing, because the input is infinite and cannot be completed at any time. Processing unbounded stream data
typically requires ingesting events in a particular sequence, such as the sequence of the events that occurred, so that the integrity of
the results can be inferred.



Bounded stream: Both the start and end of a defined stream are defined. Bounded streams can be computed after all data is read.
All data in a bounded stream can be sorted, without ingesting events in an order. Bounded stream processing is often referred to as
batch processing.

Flink Fault Tolerance

The checkpoint mechanism is an important fault tolerance method during Flink running.



The checkpoint mechanism continuously draws the snapshot of a stream application. The status



Structured Streaming is an engine to process streaming data and is built on the

RDD, Dataset, and DataFrame


In Spark 1.x, resilient distributed dataset (RDD) is a one-dimensional data set with only rows. For example,
RDD[Person] indicates that a row is a person. The person is considered as a whole in the memory.



In Spark2.x, the Dataset or DataFrame of a person is a two-dimensional data set consisting of rows and columns.
For example, a person has three columns: name: String, age: Int, and double: Double. The physical structure in
memory also explicitly displays column boundaries.

Page 72

Copyright © 2020 Huawei Technologies Co., Ltd. All rights reserved.

Program Model (1)


The core of Structured Streaming is to regard streaming data as a database table with increasing data.
Such a data processing model is similar to the data block processing model. It can apply some query
operations of the static database table to streaming data computing. Spark executes standard SQL
query statements to obtain data from an unbounded table where data is increasing.

Page 73

Copyright © 2020 Huawei Technologies Co., Ltd. All rights reserved.

Program Model (2)


Each query operation produces a result set (result table). At every triggering interval (such as 1
second), when new data is added to the table, the result table is updated. Whenever a result set has
been updated, the resulting change can be written to an external storage system (OutPut).

Page 74

Copyright © 2020 Huawei Technologies Co., Ltd. All rights reserved.

Program Model (3)


OutPut can define different storage modes:


Complete mode: The entire updated result set is written to the external storage system.
The connector of the external storage system performs the write operation on the entire
table.



Append mode: When the interval is triggered, only the new data rows in the result table
are written to the external storage. This method is applicable only to the content that
already exists in the result set (if you do not want to change the content). If the existing
data is updated, this method is not applicable.



Update mode: When the interval is triggered, only the updated data in the result table is
written to the external storage system (not available in Spark2.x).



Event time: refers to the time when an event occurs on a device. The time is
embedded before the event enters Structured Streaming, and the Structured

Stream Connection


Spark2.x supports the connection between streams. The challenge of connection
result that is generated between two data streams is that the views of the data set
are incomplete at any point in time on both sides of the connection, making it more
difficult to find matching data between the input streams. Any row received by an
input stream can be matched with any row from another future, unreceived input
stream. Therefore, for two input streams, Structured Streaming may buffer an earlier
input stream into a stream state, so as to match each new input stream with the
earlier input stream, thereby generating a connection result.

Redis Persistence - RDB


The persistence in Redis Database (RDB) mode is implemented through snapshots. When certain
conditions are met, Redis automatically takes snapshots of all data in the memory and stores the data
to a disk. By default, the data is stored in the dump.rdb file.





The time when Redis takes snapshots (in the redis.conf configuration file) is as follows:


save 900 1: The snapshot operations are performed if at least one key is changed within 900 seconds.



save 300 10



save 60 10000

Run the save or bgsave command to enable Redis to perform the snapshot operations.


The difference between the two commands is that the save command is used by the main process to perform
snapshot operations, which block other requests, and the bgsave command is used by Redis to execute the fork
function to copy a subprocess for snapshot operations.

Data Partition Management


Horizontal table partitioning for data is performed within a node based on a specified policy. The table
is divided into partitions that do not overlap within a specific range.


If you specify the PARTITION parameter when in the CREATE TABLE statement, data in the table will be
partitioned.

Partial Cluster Key


Partial Cluster Key (PCK) is the technology that is based on column storage.


It can minimize or maximize sparse indexes to quickly filter fact tables.



Only one PCK can be created in a table. A PCK can contain multiple columns, preferably no more than two
columns.



The selected columns must be restricted by simple expressions in base tables.


Such constraints are usually represented by col, op, and const. col specifies the column name, op specifies operators,
(including =, >, >=, <=, and <), and const specifies constants.

Index Design


Indexes consume storage and computing resources. Creating too many indexes has negative
impact on database performance (especially the performance of data import). Therefore,
create indexes only when they are necessary.



Page 60

Indexes are created based on the following principles:


Create indexes on columns that are frequently searched.



Create indexes on columns that are frequently connected.



Create indexes on columns that are frequently sorted.



Create indexes on columns where the WHERE clauses are frequently executed.



Create indexes on columns that are frequently aggregated and deduplicated.



Create a composite index for a query where multiple columns are joined.

Filed Design


You need to comply with the following rules to improve query efficiency when design
columns:


Use efficient data types if possible.


Use data types with high execution efficiency. Generally, the calculation efficiency of integer data is higher
than that of strings and floating point numbers (integer > floating point number > NUMREIC).



Use data types with short length, which reduces both the data file size and the memory used for computing,
improving the I/O and computing performance.



When multiple tables have logical relationships, columns with the same meaning must use the
same data type.



For string data, you are advised to use the variable-length string data type and specify the
maximum length.

Small data transmission volume:

database.



Solid security:




High development efficiency:


Dynamic Statements (1)


Dynamic statements are executed using EXECUTE IMMEDIATE and OPEN FOR.
--Pass and retrieve values (the INTO clause is used before the USING clause).
CREATE OR REPLACE PROCEDURE dynamic_proc
AS
staff_id NUMBER(6) := 200; -- Declare the staff_id variable and initialize it to 200.
first_name VARCHAR2(20); -- Declare the variable first_name.
salary NUMBER(8, 2); -- Declare the variable salary.
BEGIN
EXECUTE IMMEDIATE 'select first_name, salary from hr.staffs where staff_id = :1'
INTO first_name, salary
USING IN staff_id; -- select first_name, salary from hr.staffs where staff_id = 200;
DBMS_OUTPUT.PUT_LINE(first_name ||', '|| salary); -- Output result.
END;
-- Invoke a stored procedure.

Dynamic Statements (2)


The OPEN FOR statement is used to implement dynamic statements.
DECLARE
name VARCHAR2(20); -- Declare the variable name.
phone_number VARCHAR2(20); -- Declare the variable phone_number.
salary NUMBER(8, 2); -- Declare the variable salary.
sqlstr VARCHAR2(1024); --Declare the variable sqlstr.
TYPE app_ref_cur_type IS REF CURSOR; -- Defines the cursor type.
my_cur app_ref_cur_type; -- Define the cursor variables.
BEGIN
sqlstr := 'select first_name,phone_number,salary from hr.staffs where section_id = :1';
OPEN my_cur FOR sqlstr USING '30'; -- Open the cursor. USING is optional.
FETCH my_cur INTO name, phone_number, salary; -- Retrieve the data.
WHILE my_cur%FOUND LOOP
DBMS_OUTPUT.PUT_LINE(name||'#'||phone_number||'#'||salary);
FETCH my_cur INTO name, phone_number, salary;
END LOOP;
CLOSE my_cur; -- Close the cursor.
END;
